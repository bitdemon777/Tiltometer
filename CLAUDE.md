This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

"Tiltometer" — a tracker for home poker nights: per-player stats and money tallying.
Single-server Node.js + Express app with a vanilla-JS frontend. No framework, no
build step, no database. Most user-facing strings and code comments are in Russian;
match that when editing.

## Commands

```bash
npm install   # install express + nodemailer
npm start     # run server.js on PORT (default 3000) -> http://localhost:3000
```

There are no tests, linter, or build step. Running the app is the only verification path.

Before first run, copy `.env.example` to `.env` and set `APP_PASSWORD`. The server
reads `.env` itself via a hand-rolled parser in `server.js` (`loadEnv`) — there is no
`dotenv` dependency. Real environment variables take priority over `.env`.

## Architecture

Three layers, all in this repo:

- **`server.js`** — the entire backend. Express app serving `public/` statically plus a
  JSON `/api` surface. Persistence is the whole `data.json` file read and rewritten on
  every mutating request (`readData`/`writeData`) — fine for a few concurrent home users,
  but note there is no locking, so concurrent writers can clobber each other.
- **`public/script.js`** — the entire frontend. Vanilla JS, no modules. Renders each tab
  by setting `innerHTML` from `data` and re-attaching event listeners (`render()` dispatches
  to `renderStats`/`renderGame`/`renderHistory`/`renderPlayers`). `data` is the in-memory
  mirror of the server's `data.json`, refreshed via `loadData()` after every mutation.
- **`public/index.html` + `public/style.css`** — static shell and styling. Tab `<section>`s
  are empty containers filled by `script.js`.

### Data model (`data.json`, gitignored)

```
{ settings: { defaultBuyIn }, players: [{id, name}], events: [...] }
```

An `event` has `{id, date, buyIn, status: 'open'|'finished', participants: [...]}`.
A `participant` has `{playerId, place, rebuys, addon}`.

### Domain rules (enforced server-side, mirrored client-side)

- **Money per participant** = `buyIn + rebuys*buyIn + (addon ? buyIn : 0)`. This formula
  lives in BOTH `server.js` (`money`) and `script.js` (`moneyOf`) — keep them in sync.
- **Pot** = sum of all participants' money for the event.
- **Payout** (frontend only, `payoutOf`): pot split 50/30/20 to places 1/2/3; 1st place
  takes the remainder so prizes sum exactly to the pot. The backend does not compute payouts.
- **Stats are computed only from `finished` events**, client-side in `computeStats()`.
- Only **one `open` event** may exist at a time (enforced in `POST /api/events`).
- A `finished` event is **read-only**: cannot be edited (`PUT`) or deleted. This protects history.
- A player who has participated in any event **cannot be deleted**.

### Auth

Single shared password (`APP_PASSWORD`). `POST /api/login` validates it; every other
`/api/*` route requires the same value in the `x-app-password` header (middleware in
`server.js`). The frontend stores the password in `sessionStorage` and sends it on every
`api()` call. This is plaintext comparison — there is no per-user identity or hashing.

### Email

`POST /api/events/:id/email` sends a text summary via nodemailer/SMTP. SMTP is optional:
if `SMTP_HOST` is unset the endpoint returns an error but the rest of the app works. Port
587 implies STARTTLS, anything else (e.g. 465) implies SSL — see the `secure` flag logic.

## Conventions

- IDs are generated by `genId()` (timestamp + random base36) on the server.
- All user-supplied strings rendered into HTML go through `esc()` in `script.js` — keep
  using it when adding rendered fields to avoid XSS.
- When changing any money/payout/stats logic, update both the server and client copies and
  re-run the app to confirm pot/stats totals still reconcile.
